# Memory and Decision Log Architecture

**Last updated:** 2026-03-22
**Status:** Architecture spec for the memory layer in portfolio context.

---

## What Memory Is For

Memory in StockForge AI serves one purpose: making the portfolio agent trustworthy over time.

An agent without memory is stateless. Every decision it makes starts from scratch. It cannot learn from past proposals that the user rejected. It cannot reference past reasoning when making a similar decision. It cannot demonstrate that its behavior is consistent over time. A stateless agent is not trustworthy enough to handle money.

Memory is what allows the agent to:
- Reference past decisions when making new ones ("last month I proposed increasing NVDA, the user accepted, and it outperformed — is this situation similar or different?")
- Explain its reasoning in terms the user has already seen ("this is similar to the proposal I made on March 15")
- Avoid repeating proposals the user has consistently rejected
- Demonstrate behavioral consistency to users who are evaluating whether to trust it

Memory is also what makes the system auditable. Regulators, users, and operators all need to be able to answer the question: "why did the system do that?" Memory provides the evidence.

## Memory Components

### 1. Decision Log

The decision log is the core of the memory layer. It records every agent pipeline run and its outcome.

**Schema:**
```sql
CREATE TABLE decision_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id),
  portfolio_id UUID NOT NULL REFERENCES portfolios(id),
  pipeline_run_id UUID NOT NULL,  -- groups all records from one pipeline run
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

  -- What triggered this run
  trigger_type TEXT NOT NULL,  -- 'scheduled', 'user_request', 'portfolio_change'

  -- Inputs
  portfolio_snapshot JSONB NOT NULL,  -- portfolio state at time of run
  research_summaries JSONB NOT NULL,  -- all ResearchSummary objects
  policy_assessment JSONB NOT NULL,   -- PolicyAssessment output
  risk_report JSONB NOT NULL,         -- RiskReport output

  -- Output
  proposal JSONB,  -- RebalanceProposal (null if portfolio was healthy)
  proposal_status TEXT,  -- 'no_action_needed', 'presented', 'gate_blocked'

  -- User action (populated after user responds)
  user_action TEXT,  -- 'approved', 'rejected', 'modified', 'deferred', null
  user_action_at TIMESTAMPTZ,
  user_action_notes TEXT,

  -- Execution outcome (populated after execution completes)
  execution_id UUID REFERENCES executions(id),
  execution_status TEXT,  -- 'completed', 'failed', 'cancelled', null
  execution_at TIMESTAMPTZ
);
```

**Key properties:**
- Immutable after creation. Only `user_action*` and `execution*` fields can be updated, and only once (no overwriting).
- RLS: users can only read their own decision log.
- Operators can read all records via `/lab`.

### 2. Research Corpus

The research corpus stores documents that inform agent reasoning. It already partially exists.

**What it stores:**
- Founder notes (product decisions, strategic context, product rules)
- External references (market analyses, macro context documents)
- Research summaries (generated by the research agent, stored for reference)
- User-provided context (investment thesis documents, portfolio goals)

**Current state:** Supabase-backed, provenance-aware, with intent-aware retrieval and visibility filtering. This is a strong foundation.

**Extension needed:** Link research corpus documents to the decision log records that referenced them. This closes the provenance loop — every decision record can point to the documents that informed it.

### 3. Policy Records

Policy records store the user's current and historical constraints.

**Schema:**
```sql
CREATE TABLE portfolio_policies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id),
  portfolio_id UUID NOT NULL REFERENCES portfolios(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  effective_from TIMESTAMPTZ NOT NULL DEFAULT now(),
  effective_until TIMESTAMPTZ,  -- null = currently active

  -- Constraint definitions
  max_position_pct NUMERIC,      -- e.g., 0.10 = no single position > 10%
  max_sector_pct NUMERIC,        -- e.g., 0.30 = no sector > 30%
  prohibited_tickers TEXT[],     -- tickers that cannot be held
  required_tickers TEXT[],       -- tickers that must be held (if any)
  max_single_trade_value NUMERIC, -- max value of any single proposed trade
  automation_level TEXT,         -- 'research_only', 'propose', 'hitl', 'autonomous'

  created_by TEXT  -- 'user' or 'system_default'
);
```

**Key property:** Policy records are versioned. Changing a policy creates a new record with an `effective_from` date. The old record's `effective_until` is set to the same date. This means: every decision in the log can be evaluated against the policy that was active at the time the decision was made.

### 4. Agent Reasoning Traces

For complex multi-step reasoning (especially policy and rebalance agents), the chain-of-thought is stored separately from the structured output.

**Schema:**
```sql
CREATE TABLE agent_reasoning_traces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  decision_log_id UUID NOT NULL REFERENCES decision_log(id),
  agent_name TEXT NOT NULL,  -- 'research', 'policy', 'risk', 'rebalance'
  reasoning_text TEXT NOT NULL,  -- raw chain-of-thought
  structured_output JSONB NOT NULL,  -- the agent's final structured output
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

These traces are the most valuable audit artifact. They answer: "not just what did the agent decide, but why?"

## Retrieval Architecture

The memory layer is not just storage — it supports retrieval by the agent and by the `/lab` inspection interface.

**Retrieval modes:**

1. **Decision history retrieval:** "Show me past decisions about NVDA" — returns decision log records filtered by ticker, date range, or outcome.

2. **Similar decision retrieval:** "Find past decisions that were in a similar portfolio context" — semantic similarity search over portfolio snapshots.

3. **Research corpus retrieval:** "Find documents relevant to this portfolio analysis query" — already implemented, needs to be linked to decision log provenance.

4. **Policy audit retrieval:** "What constraints were in effect when this decision was made?" — joins decision log records to policy records by timestamp.

## Memory and the Agent's Self-Awareness

When the pipeline agent runs, it should have access to:
- The last N decision log records for this portfolio (what did the agent do recently?)
- The most recent policy assessment (is there a trend of drift in a particular direction?)
- Any proposals the user has consistently rejected (what does the user not want?)
- Research corpus context for the current holdings

This context window is what transforms the agent from a stateless function into a system with continuity. The implementation challenge is keeping this context manageable — it should be summarized and prioritized before being injected into the agent's prompt, not raw-dumped.

## Memory Integrity Requirements

1. **Decision records are append-only.** Once created, the core fields (portfolio snapshot, research summaries, proposal) cannot be modified. Only status fields can be updated.

2. **Every execution has a decision record.** If an execution occurs without a corresponding decision log entry, it is a critical integrity violation.

3. **Every decision record has a policy record.** The effective policy at the time of a decision must be reconstructable from the policy record table.

4. **Memory is not a cache.** Do not store anything in the memory layer that can be re-derived from current state. The memory layer stores what the system decided and why, not what the system currently knows.
